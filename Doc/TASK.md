### PRE-LOBBY & LOBBY FLOW
The home page (/) IS the lobby. Flow:
1. Player opens the page → sees a nickname input + "Join Queue" button.
2. On join: nickname is validated (non-empty, unique in current queue),
   player is added to the waiting list (max 4).
3. The page shows a live waiting list of connected players.
4. Any player can click "Start Game" to begin for ALL queued players.
   Start is disabled if fewer than 2 players are in queue.
5. Only ONE lobby exists at a time. If a game is in progress,
   new connections see a "Game in progress — please wait" message.
6. When the game ends (or host resets), the lobby resets and players
   can re-queue.

### GAME BOARD LAYOUT
No enforced turn order. All players see the same board state.

CENTER TABLE (always visible to all):
- Hero Deck:    face-down stack. Action: any player can request to
                draw the top card into their hand (Restricted).
- Monster Deck: face-down stack. Action: any player can reveal the
                top card → moves it to an Active Monster Slot (Restricted).
- Active Monster Slots: up to 3 face-up monster cards displayed
                simultaneously in the center. Action: any player can
                take one to their own board (Restricted).
- Discard Pile: face-up stack, only top card visible on table.
                Action: any player can view the full pile (Free).

PLAYER ZONES (one per player, 4-sided layout):
- Hand area:    cards visible ONLY to the owner (face-down stubs to
                others). Action: flip own card face-up/down (Free).
- Board area:   cards played face-up, visible to all.
                Action: take an active monster to board (Restricted).

4-SIDED CSS LAYOUT:
- Bottom: local player (no rotation)
- Top:    opponent (rotate 180deg)
- Left:   opponent (rotate 90deg)  
- Right:  opponent (rotate -90deg)
- Center: the shared table zone described above

### ACTION SYSTEM

**SETTING: Approval Mode (toggleable by any player)**
A toggle visible in the UI: "Approval Mode: ON/OFF".
- ON (default): Restricted actions require approval.
- OFF: Restricted actions execute immediately (for experienced players).
When toggled, the new state is broadcast to all clients instantly.

**FREE ACTIONS (immediate, no approval needed, always available):**
| Action           | Description                                      |
|------------------|--------------------------------------------------|
| Flip own card    | Toggle face-up/face-down on any card in own hand |
| View discard     | Open a modal showing the full discard pile        |
| Roll dice        | Roll a 1–12 die, result shown in Event Log        |

**RESTRICTED ACTIONS (require ≥1 approval from any other player,
unless Approval Mode is OFF):**
| Action              | Description                                         |
|---------------------|-----------------------------------------------------|
| Draw from Hero Deck | Take top card of Hero Deck into own hand            |
| Reveal Monster      | Flip top Monster Deck card → Active Monster Slot    |
| Take Monster        | Move an Active Monster card to own board            |
| Take from opponent  | Take a specific card from another player's hand     |
| Discard a card      | Move any own card to the Discard Pile               |
| UNDO last action    | Revert entire game state to previous snapshot       |

**APPROVAL FLOW (when Approval Mode is ON):**
1. Requesting player triggers a Restricted Action.
2. All other connected players see a popup:
   "[PlayerX] wants to [action description]. Approve / Deny"
3. Resolution rules:
   - If ≥1 player approves AND 0 players deny → action executes.
   - If ANY player denies → action is rejected immediately.
   - Timeout: 30 seconds with no response → auto-denied.
   - Only ONE pending approval request at a time (new requests are
     queued and processed sequentially).
4. Result is broadcast to all via Event Log.

**UNDO ACTION (special Restricted Action):**
- Any player can request Undo at any time.
- It reverts the ENTIRE game state to the snapshot taken before
  the last Restricted Action (history is a stack of deep-cloned
  GameState objects).
- Undo itself goes through the approval flow (Always even if Approval Mode is OFF).
- Multiple Undos in a row are possible, each reverting to the previous snapshot.

### RESTRICTED ACTION HISTORY LOG
A dedicated panel (separate from the Event Log) shows the full
chronological list of every Restricted Action executed since game
start, with format:
  [HH:MM:SS] PlayerX → ACTION_TYPE (details) [APPROVED by PlayerY]
This panel is always visible and scrollable.
An UNDO entry appears as:
  [HH:MM:SS] PlayerX → UNDO ← reverted above action [APPROVED by PlayerY]

### CLASS INTERFACES (implement exactly these signatures)
```js
class Card {
  constructor(id, name, type, path)
  // Properties: id (int), name (string), type ('hero'|'monster'),
  //             path (string), isFaceUp (bool, default false),
  //             ownerId (string|null, default null)
}

class Deck {
  constructor(type, cards[])   // type: 'hero'|'monster'
  shuffle()                    // Fisher-Yates in-place
  draw()                       // returns Card|null, removes from top
  peek(n)                      // returns top-n Cards without removing
  get size()                   // int: remaining cards
}

class Player {
  constructor(id, nickname, socketId)
  // Properties: id (string/uuid), nickname, socketId,
  //             hand[] (Card[]), board[] (Card[])
}

// GameState: plain serializable snapshot (no methods)
// Shape:
{
  players: Player[],
  heroDeck: Card[],
  monsterDeck: Card[],
  activeMonsters: (Card|null)[],  // always length 3, null = empty slot
  discardPile: Card[],
  approvalMode: bool,
  phase: 'lobby' | 'playing' | 'ended'
}

class ApprovalQueue {
  // Manages one pending action at a time.
  // Methods: enqueue(action), approve(actionId, playerId),
  //          deny(actionId, playerId), timeout(actionId)
  // Emits resolution events via the io instance passed to constructor.
}

class GameManager {
  constructor(io, cards[])
  startGame(players[])
  getState()                          // deep-clone of current state
  getStateForPlayer(playerId)         // getState() with other players'
                                      // hands replaced by face-down stubs
  pushHistory()                       // snapshot before each Restricted Action
  undo()                              // pop & restore last snapshot
  requestAction(action, requesterId)  // route to approvalQueue or execute
  toggleApprovalMode(playerId)        // flip approvalMode bool, broadcast
  addToRestrictedLog(entry)           // append to restricted history
  getRestrictedLog()                  // return full array
}
```

### SOCKET.IO EVENT CONTRACT

**Client → Server:**
| Event               | Payload                                      |
|---------------------|----------------------------------------------|
| `join_lobby`        | `{ nickname }`                               |
| `start_game`        | `{}`                                         |
| `request_action`    | `{ type, payload }` (see action types above) |
| `respond_approval`  | `{ actionId, decision }` (bool)              |
| `flip_card`         | `{ cardId, zone }` ('hand'\|'board')         |
| `view_discard`      | `{}`                                         |
| `roll_dice`         | `{}`                                         |
| `toggle_approval`   | `{}`                                         |
| `reset_lobby`       | `{}` (returns to lobby state)                |

**Server → Client:**
| Event               | Payload                                           |
|---------------------|---------------------------------------------------|
| `lobby_update`      | `{ players: [{id,nickname}], gameInProgress }`    |
| `game_start`        | `{ yourPlayerId }`                                |
| `state_update`      | `GameState filtered via getStateForPlayer()`      |
| `approval_request`  | `{ actionId, type, requesterNickname, details }`  |
| `approval_result`   | `{ actionId, granted, approverNickname }`         |
| `event_log`         | `{ message, timestamp }`                          |
| `restricted_log`    | full restricted action history array              |
| `dice_result`       | `{ nickname, result }`                            |
| `error`             | `{ message }`                                     |

**PRIVACY RULE (mandatory):** When sending `state_update`, other
players' `hand` arrays must be replaced with stub objects:
`{ id: card.id, isFaceUp: false, ownerId: card.ownerId }`
Never send actual card data (name, path) for another player's
face-down hand cards.
